## 核心问题

### 问题1: 分数合并逻辑错误 (P0)

**位置**: `src/lib/rag.ts:558-599`

**问题**: 模糊搜索的 `score` 和向量搜索的 `similarity` 来源不同、量纲不同，直接合并不合理。

---

### 问题2: 缺少 BM25 稀疏搜索 (P0)

当前模糊搜索是简单的字符串匹配，BM25 算法能：
- 考虑词频（TF）和逆文档频率（IDF）
- 对常见词降权，稀有词升权
- 与向量搜索形成语义互补

**BM25 vs Fuzzy Search**：
- BM25：精确分词 + 统计，区分词重要性
- Fuzzy Search：字符串模糊匹配，容错、部分匹配
- 两者互补，不互相替代

---

### 问题3: 分块策略不够智能 (P1)

**位置**: `src/lib/rag.ts:92-171`

- 仅基于 `\n\n` 分割段落
- 未考虑 Markdown 结构（标题 `##`、代码块 ` ``` `、列表等）
- 可能破坏语义完整性

---

### 问题4: 元数据不足 (P1)

**位置**: `src/db/vector.ts:4-11`

缺少的关键元数据：
- 文档路径/目录结构
- 标题层级信息
- 章节标题/关键词

---

### 问题5: 缺少上下文扩展 (P2)

当前返回的检索结果是独立的 chunk，缺少：
- 句子窗口扩展
- 父文档上下文
- 相邻 chunk 关联

## 第一阶段：检索优化（不影响现有向量）

> ✅ 这些优化只改检索逻辑，**现有向量数据完全不受影响**，可以立即实施

### Phase 1: 修复分数合并逻辑 (P0)

**目标**：将不同来源的分数归一化后再合并

**改动文件**：`src/lib/rag.ts`

**主要工作**：
1. 新增 `normalizeScore` 函数，对 fuzzy/vector/bm25 分数分别归一化
2. 新增 `calculateHybridScore` 函数，支持可配置的权重合并
3. 修改 `getContextForQuery` 函数，收集所有检索结果后统一计算混合分数

**新增配置项**：
- `ragFuzzyWeight`: 模糊搜索权重（默认 0.2）
- `ragVectorWeight`: 向量搜索权重（默认 0.7）
- `ragBm25Weight`: BM25 权重（默认 0.1）

**预期效果**：修复分数量纲不一致的问题，使多种检索方式的结果能够公平比较

---

### Phase 2: 添加纯 TS 实现的 BM25 检索 (P0)

**目标**：引入 BM25 算法，与模糊搜索、向量搜索形成互补

**改动文件**：
- 新建：`src/lib/bm25.ts`
- 修改：`src/lib/rag.ts`

**为什么自己实现？**

| 对比项 | MiniSearch / wink-bm25 | 自己实现 |
|:---|:---|:---|
| 中文支持 | ❌ 需要额外分词库 | ✅ 原生支持 |
| 额外依赖 | nodejieba (5-10MB) | 无 |
| 总体积 | ~6-15MB | ~3KB |
| 可控性 | 受限于库API | 完全可控 |

**中文分词策略**：

采用混合分词策略，简单但有效：

1. **按边界分割**：标点、空格、中英文边界
2. **过滤单字**：保留 2+ 字词语（中文大部分词 ≥ 2 字）
3. **过滤数字**：纯数字不参与匹配

**示例**：
```
输入: "RAG检索增强生成系统用于智能问答"
输出: ["RAG", "检索", "增强", "生成", "系统", "用于", "智能", "问答"]
```

**效果**：
- 自动过滤单字虚词（的、是、了...）
- 保留专业术语（RAG、LLM、API...）
- 无需外部词库
- 覆盖 80% 常见场景

**主要工作**：
1. **新建 BM25Index 类** (`bm25.ts`)
   - 实现 `tokenize` 方法：中文友好的分词
   - 实现 `index` 方法：构建文档索引
   - 实现 `calculateIDF` 方法：计算逆文档频率
   - 实现 `search` 方法：返回相关文档排序
   - 支持 `update` 和 `delete` 操作

2. **集成到 RAG** (`rag.ts`)
   - 添加 `initBM25Index` 函数，基于现有文件内容构建索引
   - 在 `getContextForQuery` 中添加 BM25 检索路径
   - BM25 结果与 fuzzy/vector 结果一起参与分数合并

**说明**：BM25 是独立索引，基于原始文件文本，**不影响现有向量数据库**

**预期效果**：增加基于词频统计的精确检索能力，区分常见词和稀有词的重要性

---

### Phase 3: Markdown 感知的智能分块 (P1)

> ⚠️ **第二阶段改动**：此 Phase 改变分块策略，需要重新计算向量。建议先完成第一阶段，再考虑是否需要此优化。

**目标**：按 Markdown 结构（标题、代码块）智能分块，保持语义完整性

**改动文件**：`src/lib/rag.ts`

**主要工作**：
1. **新增 `MarkdownSection` 类型**：定义章节结构
2. **新增 `parseMarkdownStructure` 函数**：
   - 解析标题层级（`#` 到 `######`）
   - 识别并保持代码块完整性
   - 返回章节树结构

3. **新增 `splitLargeSection` 函数**：
   - 对过大的 section 进行二次分块
   - 避开代码块，按段落分割
   - 保持代码块完整

4. **重构 `chunkText` 函数**：
   - 优先使用 Markdown 结构感知分块
   - 降级到原有简单分块逻辑

**预期效果**：分块更符合文档语义结构，向量嵌入更精确，但需要用户重新处理所有文件

---

### Phase 4: 增强向量元数据 (P1)

> ⚠️ **第二阶段改动**：此 Phase 需要数据库迁移，通常与 Phase 3 配合使用。

**目标**：扩展向量文档表结构，添加更多元数据字段

**改动文件**：
- `src/db/vector.ts`
- `src/lib/rag.ts`

**主要工作**：
1. **扩展 `VectorDocument` 接口**：
   - 新增 `filepath`：完整文件路径
   - 新增 `title`：所属章节标题
   - 新增 `heading_level`：标题级别
   - 新增 `chunk_type`：分块类型
   - 新增 `prev_chunk_id` / `next_chunk_id`：相邻块ID
   - 新增 `metadata`：其他元数据JSON

2. **数据库迁移**：
   - 添加新列（使用 `ALTER TABLE`）
   - 创建组合索引
   - 处理现有数据的兼容性

3. **修改 `processMarkdownFile`**：
   - 调用 `parseMarkdownStructure` 获取标题信息
   - 保存向量时附带元数据

**预期效果**：支持更精细的检索和过滤，但需要数据库迁移

---

### Phase 5: 句子窗口检索 (P1)

> **说明**：检索时动态扩展上下文，**不需要改动现有向量数据**。

**目标**：返回匹配 chunk 的上下文窗口，提供更完整的语义

**改动文件**：`src/lib/rag.ts`

**主要工作**：
1. **新增 `expandWithSentenceWindow` 函数**：
   - 输入：原始检索结果、窗口大小
   - 逻辑：对每个结果，获取同一文件的前后 N 个 chunk
   - 输出：合并后的窗口内容，标记中心匹配位置

2. **修改 `getContextForQuery`**：
   - 在向量搜索结果返回后调用扩展函数
   - 获取更多候选结果（因为窗口会合并）

**新增配置项**：
- `ragWindowSize`: 窗口大小（默认 2，即前后各取 2 个 chunk）

**预期效果**：返回的上下文更完整，AI 回答质量更好

---

### Phase 6: 查询转换优化 (P2)

> **说明**：只改变查询方式，生成多个查询变体，**不需要改动现有向量数据**。

**目标**：通过查询扩展和同义词替换提高召回率

**改动文件**：`src/lib/rag.ts`

**主要工作**：
1. **定义同义词词典**：
   - 技术术语：ai/llm/rag 等
   - 通用词汇：如何/怎么/怎样 等
   - 支持配置化扩展

2. **新增 `transformQuery` 函数**：
   - 输入：原始查询、配置
   - 逻辑：生成多个查询变体（同义词替换）
   - 输出：查询变体数组

3. **新增 `expandWithSynonyms` 函数**：
   - 基于同义词词典生成变体
   - 避免生成重复查询

4. **修改 `getContextForQuery`**：
   - 对每个关键词生成查询变体
   - 对每个变体执行检索
   - 合并所有结果

**新增配置项**：
- `ragEnableQueryExpansion`: 是否启用查询扩展（默认 true）
- `ragMaxQueryVariations`: 最大变体数量（默认 3）

**预期效果**：通过多路召回提高检索覆盖率

### BM25 vs Fuzzy Search 补充说明

两者**互补而非替代**关系：

| 特性 | BM25 | Fuzzy Search |
|:---|:---|:---|
| 匹配方式 | 精确分词 + 统计 | 字符串模糊匹配 |
| 核心能力 | 区分词重要性（IDF） | 容错、部分匹配 |
| 适用场景 | 精确关键词检索 | 拼写错误、模糊查询 |

**结论**：保留 fuzzy_search，同时引入 BM25，与向量搜索形成三路互补。
